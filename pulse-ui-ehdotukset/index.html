Mobile app plan

PULSE is not a survey. It is a lightweight signal at the right moment.

UX starting point: the attendee should not stop to fill in a form in the middle of an event. PULSE keeps the first action to one tap and asks for more detail only after the user has already expressed something.

Minimize frictionThe first reaction happens directly in the session view. No separate survey path before interest has been shown.
Keep the contextEach reaction is always stored with the programme item, time and user role so the signal does not become detached.
Return valueThe attendee sees where their PULSE went: personal feed, saved highlight or a ready-to-share summary.

Required view 01

Session view where PULSE is part of the programme.

This is the most important entry point. PULSE should not feel like a separate research task, but like a micro-action exactly where the user experiences the content.

Why below the programme card?The user automatically connects the reaction to the right content. The organizer does not have to infer later what the feedback referred to.
Why four large buttons?At an event, usage is fast and often one-handed. Large buttons reduce mis-taps and keep the choice clear.
Why show “1 tap”?It tells the user this is not a long survey. That will likely increase usage.
Why keep the bottom nav?PULSE does not trap the user in a mode. They can return to the feed, profile or home just like in other app views.
UX decision: the first PULSE is saved immediately on tap. The extra reason is the next step, not a condition for saving.

Required view 02

Optional reason layer as a bottom sheet.

A thumb alone says little. The reason layer makes PULSE useful, but it has to stay lightweight. That is why the view opens after the reaction and does not replace the main experience.

Why a bottom sheet?The user stays in the same context. The dialog feels like a continuation of the choice, not a separate page.
Why predefined reason chips?Most people do not write during an event. Chips provide better-quality data with one extra tap.
Why include an open field?When an attendee has a real observation, it should not be forced into a preset category. The field is an extra, not the main action.
Why is Skip visible?Forcing reduces trust. Skipping keeps the original reaction valuable and preserves the flow.
UX decision: the reason layer is progressive detail. A good default is to close it automatically after saving and return the user to the session view.

Required view 03

Confirmation that gives value back to the user.

If PULSE feels only like data collection for the organizer, usage easily becomes occasional. The confirmation view shows that the reaction was saved and gives the user a useful next action.

Why “saved to feed”?The attendee gets personal value: the highlight can be found later and the moment does not disappear into the event flow.
Why two CTAs?The primary path is to continue the programme. The value-add path is to view or edit the personal feed highlight.
Why not ask another question immediately?Too much questioning feels tiring. At this point the UI should reward, not ask for more.
Why an “answered” state?The same user should not get the same PULSE request again. This prevents irritation and duplicate data.
UX decision: after each PULSE, show a clear feedback state. The default action is to return to the event quickly.

Required view 04

Personal feed where PULSE becomes a memory trace.

PULSE is useful for the attendee too if it helps remember and continue event insights. The feed turns a reaction into a personal tool.

Why a personal feed?The attendee can return to what felt important. This strengthens the app’s value after the event.
Why “Saved / Shareable” tabs?Not every reaction is shareable. Tabs separate personal memory traces from content meant to travel outward.
Why a list item structure?It is a familiar mobile element from the template and scales to many PULSEs without a heavy card flood.
Why copy/edit?“Share” needs control. The user does not want to send an AI summary without checking it.
UX decision: the personal feed is the anchor for post-event use. Without it, PULSE remains a momentary reaction.

Required view 05

Organizer view where the signal is interpretable.

The organizer does not need just a counter. A good PULSE view shows what content the reactions relate to, what theme is rising and whether something requires action.

Why report cards at the top?The organizer immediately gets the situation: what is happening now and what is unusual.
Why progress indicators?They are fast enough to read live. Deeper analysis can come later in the report.
Why theme lists?This is where the reason layer creates value: feedback becomes themes that guide follow-up action.
Why an insight toast?The organizer should be shown anomalies, not forced to find them in raw data.
UX decision: the organizer view should separate “what happened” from “what should be done next”.

Required view 06

States that make the feature reliable.

Small states define the experience: has PULSE already been given, can it still be given, what happens with poor connectivity and how can the user return later.

Answered statePrevents repeated asking and makes saving visible.
TimingPULSE should only appear at a relevant moment. Asking too early is worthless.
Offline supportConnectivity varies at event venues. The user should not lose a reaction for technical reasons.
Sharing laterSharing requires consideration. The feed makes it possible to return without pressure.
UX decision: PULSE needs at least idle, active, selected, saved, answered and offline-pending states before implementation.

Core implementation recommendation

First releaseBuild Views 01–03 first: session PULSE, optional reason layer and confirmation.
Do not make this a surveyIf the user has to open a separate feedback page, the feature loses its strength.
Data designStore reaction, session, time, role/profile, optional reason and state as separate fields.
Next prototypeClickable flow: reaction → reason sheet → saved → personal feed → organizer signal.
